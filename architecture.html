🏏 Cricket Rules Search

A natural-language search engine for cricket rules — built entirely on free tiers. This page explains the architecture, end to end, with every step and every flow.

1 · The big idea

A plain chatbot has two problems: it doesn't know your specific league rules, and it can confidently invent answers. We solve both with RAG — an "open-book exam" for the AI.

A

You ask a question in plain English

“If the batter is out of the crease and the keeper breaks the stumps, is it run out or stumped?”

B

The system searches your real rule documents

It finds the 3–5 passages most relevant by meaning, not by keyword.

C

It hands those passages to the AI with strict instructions

“Answer using only these rules. Cite the source and rule number. If they don't cover it, say so.”

D

You get a grounded, cited answer

Backed by a real rule — never a hallucination.

2 · The three free layers

The app is split into three independent pieces. This separation is what makes it secure, maintainable, and free.

The Face

GitHub Pages

Static HTML / CSS / JS. Just the search box and the rendered answer. Knows nothing secret.

FREE
The Brain

Hugging Face Space

Python backend (FastAPI). Searches the rules, builds the prompt, holds the secret key, calls the AI.

FREE TIER
The Writer

Groq LLM

A fast large language model (e.g. Llama 3.3 70B) that writes the final natural-language answer.

FREE TIER
Why split it? ① Security — the secret API key lives only on the backend, never in the browser. ② Separation of concerns — the pretty part and the smart part can change independently.

3 · Build-time step (done once, offline)

Before anyone can search, we prepare the rules once. This is the ingest.py script. Re-run it only when the rules change.

Input
Raw rule documents
BCMCL ICC MCC — plain text in data/raw/ (extracted from your PDFs).
① read & chunk
Step 1 — Chunk
Split into small logical units
One rule / sub-clause per chunk. This is the single biggest lever on quality — long blocks dilute meaning.
② tag metadata
Step 2 — Tag
Label every chunk
{ source: "BCMCL", rule_id: "12.3", text: "..." }
③ embed
Step 3 — Embed
Turn each chunk into a vector
Using all-MiniLM-L6-v2 (runs locally, no API). Each chunk becomes a list of numbers representing its meaning.
④ save
Output
A single index file
data/index/rules_index.npz — vectors + metadata, committed to the repo. The backend loads this on startup.
Why no vector database? The rule set is tiny (3 documents). We embed everything once and load it into memory. Cosine similarity with NumPy is enough — no database to host, nothing to get wiped on restart, one fewer moving part.

5 · How "search by meaning" actually works

This is the clever core — and why it feels smart instead of like Ctrl+F.

Embeddings = meaning as numbers

A model converts each piece of text into a vector (a long list of numbers) that captures its meaning. Texts about similar ideas get similar vectors — even with zero shared words.

"batter out of crease, keeper breaks stumps"  →  [0.21, -0.04, 0.88, ...]
"stumping dismissal"                          →  [0.19, -0.07, 0.85, ...]   ← close!
"boundary scored over the rope"               →  [-0.55, 0.62, 0.10, ...]   ← far away

Cosine similarity = "how close in meaning?"

To find matches, we measure the angle between the question-vector and each rule-vector. A small angle (score near 1.0) means very similar meaning; near 0 means unrelated. We keep the top few.

question  ↘
           ↘  small angle  →  score 0.91  ✅ keep
   rule A  ───────────────

question  ↘
           ↘  wide angle   →  score 0.12  ❌ drop
   rule B  ╱
The shortcut that keeps it free: with only 3 documents we never need a hosted vector database — all vectors sit in memory and a few lines of NumPy do the math instantly.

6 · Source priority (BCMCL → ICC → MCC)

Local league rules win; international rules are the fallback. We do not force this into the math — we let the LLM decide, guided by instructions.

1

Retrieve across all three sources together

So cross-references still work and nothing relevant is hidden.

2

Pass each chunk's source label into the prompt

The model can see whether a passage is BCMCL, ICC, or MCC.

3

Instruct the model on priority

See the exact wording below.

"Answer using the retrieved rules. If a BCMCL rule covers the
question, base the answer on BCMCL. Otherwise use ICC. If neither
applies, use MCC. Always state which source and rule number the
answer comes from. If none of the retrieved rules answer the
question, say so — do not invent a rule."
This keeps priority logic simple, allows cross-referencing, and blocks hallucinated rules — all at once.

7 · Security & the secret key

The golden rule: the API key never touches the browser.

🔒

Key lives only as a secret in the Hugging Face Space settings

The frontend calls your backend; only your backend calls Groq.

🌐

CORS

The backend allows requests only from your GitHub Pages domain.

Why it matters: if the key were in the browser's JavaScript, anyone could view-source, copy it, and burn your free quota (or worse). Proxying through the backend is the standard, professional fix.
